`docker inspect --format '{% raw %}{{.State.Health.Status}}{% endraw %}'` came back empty. Not `unhealthy`, not `starting` — nothing, as if the container had never had a `HEALTHCHECK` in the first place. I was looking at a `tor` container whose Dockerfile declares one in plain sight, running an image pulled straight from our own registry a few minutes earlier. The obvious hypothesis formed immediately: something in our push pipeline is stripping the field.

## The hypothesis, and why it looked right

Every Linux image in this fleet gets pushed the same way: `docker buildx bake --push`, on the `docker-container` driver. (Windows builds are the one exception — buildx bake doesn't support them, so they go through a separate `--load` + plain `docker push` path with a different mediatype outcome, more on that below.) Nothing in the bake command asks for attestations — but that's not the same as attestations being off. BuildKit adds a minimal provenance attestation to every push by default on this driver, no flag required. That default changes how an image gets packaged under the hood — new manifests, new mediatypes, new layers of indirection. "One of those side effects is eating the Healthcheck field" was a reasonable first guess, and it fit the evidence: `sslh`, built through the identical pipeline, showed the exact same symptom. Fleet-wide, one shared cause.

## The test that told me I was wrong (it wasn't)

Before writing that up as a bug report, I built a `HEALTHCHECK`-bearing Dockerfile twice — once with the default provenance attestation left on, once with `--provenance=false` — and diffed the resulting image configs. They came back identical, `Healthcheck` present in both. That result reads as "nothing in the build is throwing the field away," and I published on that basis: the field survives, something downstream just isn't reading it.

Going back to check that conclusion against a live, currently-published image, it doesn't hold up. `skopeo inspect --config` on `ghcr.io/oorabona/tor:latest` — fetched straight from the registry, no local daemon in the path — has no `Healthcheck` key at all. Not present-but-unread. Absent. I checked `sslh` too, specifically because its `HEALTHCHECK` is a single trivial `busybox nc -z` call, about as far from `tor`'s multi-line shell script as a healthcheck gets — if this were some Dockerfile-parsing edge case tied to script complexity, a one-liner should be immune. It isn't: same gap, same nothing.

So the field genuinely doesn't survive into what this pipeline publishes, for both containers, regardless of how simple or complicated the check itself is. My own diff test said otherwise, and I don't have a clean explanation for why the two disagree — I can't reproduce BuildKit's default-attestation behavior in the sandbox I did this follow-up investigation in, so I couldn't re-run the original test to find where it diverges from reality. What I can say is which mechanisms I ruled out: it isn't `crane flatten` (the fleet does run that on the *other*, non-bake build path, but `tor`'s published image still has its original three layers, not the single flattened layer that tool would leave behind, and bake-built images never go through it in this pipeline); it isn't a multi-stage Dockerfile losing the instruction on the way to a later `FROM` (`tor`'s Dockerfile is single-stage); and it isn't the healthcheck script's complexity (the `sslh` one-liner rules that out). What's left, unconfirmed, is the most boring explanation: BuildKit's own OCI-mediatype config writer, on this driver, doesn't carry the field through. I looked for a direct citation for that specific behavior and didn't find one — so take it as my best current guess, not a verified mechanism.

## What OCI actually says about Healthcheck

`Healthcheck` isn't entirely absent from the OCI spec — [config.md](https://github.com/opencontainers/image-spec/blob/v1.1.1/config.md) lists it as an optional property, but marks it *reserved*, and the [compatibility matrix](https://github.com/opencontainers/image-spec/blob/v1.1.1/media-types.md) is blunter about what that means in practice: "`.config.Healthcheck`: only present in Docker, and reserved in OCI." Docker's own separate spec, [`docker-image-spec`](https://github.com/moby/docker-image-spec), models the field as a genuine extension bolted onto the base OCI config type — which tells me the field *can* be carried in an OCI-shaped config, by tools that choose to. Whatever BuildKit's OCI exporter is doing on this driver, it isn't required to carry a field the base OCI schema doesn't define semantics for, and on the evidence above, it doesn't.

That's also consistent with the opening symptom: the container wasn't showing `starting` or `unhealthy`, it was showing nothing, because the Docker daemon instantiates a container's healthcheck from the pulled image's own config — if the field was never in that config to begin with, there's nothing to reach for. The instruction survives only as a cosmetic string in the image's build history, which nothing reads at runtime.

## Confirming there's no third option

The instinct here is to go looking for the flag that fixes it. I did — and the upstream trail is illuminating, if not quite as tidy as I first assumed. [docker/buildx#3047](https://github.com/docker/buildx/issues/3047) asks BuildKit's maintainers directly whether attestations could be left out of the image index, and gets a clear answer: no, by design — attestations stay in the index so other tooling can find them, and "BuildKit does not support any other distribution method for attestations that is outside of the build result." No opt-out, closed as working as intended. [moby/buildkit#6070](https://github.com/moby/buildkit/issues/6070) is a separate, still-open request asking BuildKit to actually document this mediatype coupling somewhere instead of leaving people to rediscover it. And [opencontainers/image-spec#749](https://github.com/opencontainers/image-spec/issues/749) is the spec project's own tracking issue asking for OCI-defined *semantics* for that reserved slot — not a new field, since the schema already reserves the name, but an actual behavior spec-conformant tools would be obligated to honor. Open since 2018, still unresolved.

The choice is genuinely binary: drop the default provenance attestation and get Docker mediatype (and, on the evidence here, the `Healthcheck` field) back, or keep it and accept the field is gone from what's published. There's no flag combination that gets both. What makes this an easy call is that it isn't really a deliberate trade this fleet made — nobody opted into BuildKit's default provenance attestation, it's just what the `docker-container` driver does unless you explicitly turn it off with `--provenance=false` or `BUILDX_NO_DEFAULT_ATTESTATIONS=1`. Fighting a free default to restore a field almost nothing in our own tooling checks isn't worth the churn — especially next to what that default is quietly buying us: real, if currently unsigned, provenance metadata on every image, ahead of the signed `actions/attest-build-provenance` step already on the roadmap to formalize it. SBOM attestation is a separate, already-signed mechanism (`actions/attest-sbom`, out-of-band via the GitHub Attestations API) — it doesn't touch the image's own manifest index and isn't part of this trade-off at all.

## Checking the actual damage

A fleet-wide root cause doesn't automatically mean fleet-wide breakage, and it's worth checking rather than assuming. Our own end-to-end tests already tolerate a missing health status — they were written defensively before this was ever diagnosed, which is a coincidence I'll take. Of the stacks in this repo that gate a service on `depends_on: condition: service_healthy`, most already define their own Compose-level `healthcheck:`, which overrides whatever the image itself declares — unaffected by any of this. One stack didn't have that override and would have genuinely failed to start. That's a real, narrow bug, and it got its own fix.

## What I'd tell past-me

- **A test that "kills a theory" only kills it if you can trust the test.** My first diff said the field survives; the live artifact says it doesn't. I published the first answer before checking it against the thing readers would actually pull. Test against production, not just against a local repro of it.
- **Ruling out causes is progress even when you can't name the real one.** I don't have a citation for exactly why BuildKit's OCI writer drops this field on this driver — but I do know it isn't the flatten step, isn't a multi-stage Dockerfile issue, and isn't healthcheck-script complexity, because I checked each one against the real artifact. An honest "I ruled out A, B, and C" beats a confident guess dressed up as a finding.
- **A shared symptom across two containers is evidence of a shared cause — it's not evidence of *which* cause, and it's not proof the cause is what your first test says it is.** Both `tor` and `sslh` losing the field the same way is what let me rule out per-container explanations at all; it took going back to the registry twice to find out my first "which cause" answer was wrong.
- **A root cause confirmed doesn't mean the blast radius is what you assumed.** The honest next question after "found it" is "so what actually breaks," not "how do I fix it everywhere."
- **Some findings are correctly closed as "won't fix."** Not every root cause has a repair; some are the visible edge of a default that isn't worth fighting to reverse, even when the exact mechanism stays partly unresolved.
