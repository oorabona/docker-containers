`docker inspect --format '{% raw %}{{.State.Health.Status}}{% endraw %}'` came back empty. Not `unhealthy`, not `starting` — nothing, as if the container had never had a `HEALTHCHECK` in the first place. I was looking at a `tor` container whose Dockerfile declares one in plain sight, running an image pulled straight from our own registry a few minutes earlier. The obvious hypothesis formed immediately: something in our push pipeline is stripping the field. I was wrong about that — and wrong, briefly, about why I was wrong, before a second look sorted it out.

## The hypothesis, and why it looked right

Every `latest`-tagged Linux image in this fleet gets pushed the same way: `docker buildx bake --push`, on the `docker-container` driver. (Two kinds of build stay off that path and go through a separate `--load` + plain `docker push` instead, with a different mediatype outcome: Windows builds, since buildx bake doesn't support them, and retained — non-latest — version tags for containers that keep older versions on the flat matrix rather than the bake DAG.) Nothing in the bake command asks for attestations — but that's not the same as attestations being off. BuildKit adds a minimal provenance attestation to every push by default on this driver, no flag required. That default changes how an image gets packaged under the hood — new manifests, new mediatypes, new layers of indirection. "One of those side effects is eating the Healthcheck field" was a reasonable first guess, and it fit the evidence: `sslh`, built through the identical pipeline, showed the exact same symptom. Fleet-wide, one shared cause.

## The test that said otherwise

Before writing that up as a bug report, I built a `HEALTHCHECK`-bearing Dockerfile twice — once with the default provenance attestation left on, once with `--provenance=false` — and diffed the resulting image configs. They came back identical, `Healthcheck` present in both, word for word. That result reads as "nothing in the build is throwing the field away." Which meant the actual question wasn't "why is this missing" — it was "why does it look missing when it isn't."

## A second test that briefly said the opposite

Wanting to confirm that against a live, currently-published image rather than a local repro, I ran `skopeo inspect --config` against `ghcr.io/oorabona/tor:latest` on the real registry. No `Healthcheck` key. Ran the same against `sslh` — deliberately chosen because its healthcheck is a single trivial `busybox nc -z` call, about as far from `tor`'s multi-line shell script as it gets, to rule out a per-Dockerfile parsing quirk. Same gap. Two containers, wildly different healthcheck complexity, identical absence. For a while I believed this contradicted my own earlier diff test, and started rewriting this post around a different conclusion: that the field genuinely doesn't survive into what gets published, and my first test just hadn't been representative of the real pipeline.

It hadn't occurred to me to check the tool doing the checking. `skopeo inspect --config`, without `--raw`, doesn't hand back the registry's actual bytes — it deserializes the config into skopeo's own internal representation and re-serializes it for display, and that representation is typed to the OCI-standard config shape, the same one that has no slot for `Healthcheck`. Add `--raw` and the same command against the same live `tor` image returns the config with `Healthcheck` fully intact. My "second, more careful" test had walked straight into the exact bug it was trying to characterize: a tool that understands the base OCI schema and quietly drops a field outside it. I'd nearly published a correction to my own correct answer, on the strength of a tool doing precisely the thing the post is about.

## What's actually happening

The gap is entirely in how something *reads* the config, not in what got written — and by this point I had two independent, hands-on examples of exactly that read-side gap, not just one inference. `skopeo inspect --config` (no `--raw`) is one: it silently omits a field its own OCI-typed struct doesn't model. `docker inspect --format '{% raw %}{{.Config.Healthcheck}}{% endraw %}'` on this same image, run through this fleet's local Podman/Docker-CLI-compatible tooling, is the other — it doesn't return an empty value, it throws a template error: `can't evaluate field Healthcheck in type *v1.ImageConfig`. That's not a value that's missing; it's a struct that was never given a field for it in the first place, on the reading side.

A provenance attestation is itself a small manifest, and it has to live somewhere alongside the image it's describing. The place BuildKit puts it is an OCI image index: a container of manifests, where the real image and its attestation sit as siblings — [Docker's own docs](https://docs.docker.com/build/metadata/attestations/) confirm attestations "attach to images as a manifest in the image index." That structural requirement pulls the image's own manifest into OCI mediatype along with everything else in the index — confirmed by testing `oci-mediatypes=false`, which does nothing to stop it; once attestations are on, OCI mediatypes aren't optional.

`Healthcheck` isn't entirely absent from the OCI spec, either — [config.md](https://github.com/opencontainers/image-spec/blob/v1.1.1/config.md) lists it as an optional property, but marks it *reserved*, and the [compatibility matrix](https://github.com/opencontainers/image-spec/blob/v1.1.1/media-types.md) is blunter about what that means in practice: "`.config.Healthcheck`: only present in Docker, and reserved in OCI." The name is in the schema; the semantics aren't — nothing in the spec tells a reader what to *do* with that field, so a spec-conformant reader is free to carry it through without ever acting on it, or to type its own config parsing against the OCI-only shape and never see the field at all. Docker's own separate [`docker-image-spec`](https://github.com/moby/docker-image-spec) models `Healthcheck` as a genuine extension bolted onto the base OCI type, which is exactly why this is a per-tool question rather than a yes/no one: a tool that parses against the extended type sees it, a tool that parses against the base OCI type doesn't, and both are reading the identical bytes.

That's also the answer to the question the opening symptom actually asked: the container wasn't showing `starting` or `unhealthy`, it was showing nothing, because the Docker daemon instantiates a container's healthcheck from the pulled image's own config, through whichever of those two shapes its own image-handling code uses — and on an OCI-mediatype pull, that path evidently doesn't reach for the extension. The instruction survives only as a cosmetic string in the image's build history, which nothing reads at runtime.

Nothing was stripped. Multiple independent readers were each doing exactly what a strictly OCI-typed reader should do with a field the base schema reserves but never defines: not look for it.

## Confirming there's no third option

The instinct here is to go looking for the flag that fixes it. I did — and the upstream trail is illuminating, if not quite as tidy as I first assumed. [docker/buildx#3047](https://github.com/docker/buildx/issues/3047) asks BuildKit's maintainers directly whether attestations could be left out of the image index, and gets a clear answer: no, by design — attestations stay in the index so other tooling can find them, and "BuildKit does not support any other distribution method for attestations that is outside of the build result." No opt-out, closed as working as intended. [moby/buildkit#6070](https://github.com/moby/buildkit/issues/6070) is a separate, still-open request asking BuildKit to actually document this mediatype coupling somewhere instead of leaving people to rediscover it. And [opencontainers/image-spec#749](https://github.com/opencontainers/image-spec/issues/749) is the spec project's own tracking issue asking for OCI-defined *semantics* for that reserved slot — not a new field, since the schema already reserves the name, but an actual behavior every spec-conformant tool would be obligated to honor, rather than leaving it to whichever config type a given tool happened to parse against. Open since 2018, still unresolved.

Fixing this fleet-wide would mean either dropping the default provenance attestation, or getting every tool in the chain — including ones we don't control, like `docker inspect` itself — to parse against Docker's extended config type instead of the OCI-only one. Neither is a flag we can flip. What makes giving up on it an easy call is that it isn't really a deliberate trade this fleet made — nobody opted into BuildKit's default provenance attestation, it's just what the `docker-container` driver does unless you explicitly turn it off with `--provenance=false` or `BUILDX_NO_DEFAULT_ATTESTATIONS=1`. Fighting a free default to restore a field almost nothing in our own tooling checks isn't worth the churn — especially next to what that default is quietly buying us: real, if currently unsigned, provenance metadata on every image, ahead of the signed `actions/attest-build-provenance` step already on the roadmap to formalize it. SBOM attestation is a separate, already-signed mechanism (`actions/attest-sbom`, out-of-band via the GitHub Attestations API) — it doesn't touch the image's own manifest index and isn't part of this trade-off at all.

## Checking the actual damage

A fleet-wide root cause doesn't automatically mean fleet-wide breakage, and it's worth checking rather than assuming. Our own end-to-end tests already tolerate a missing health status — they were written defensively before this was ever diagnosed, which is a coincidence I'll take. Of the stacks in this repo that gate a service on `depends_on: condition: service_healthy`, most already define their own Compose-level `healthcheck:`, which overrides whatever the image itself declares — unaffected by any of this, since Compose evaluates its own override directly rather than trying to resolve the image's. One stack didn't have that override and would have genuinely failed to start. That's a real, narrow bug, and it got its own fix.

## What I'd tell past-me

- **The tool you're using to verify a claim can have the exact same blind spot you're investigating.** I ran `skopeo inspect --config`, got a negative result, and nearly rewrote this whole post around it — without noticing that tool was itself parsing against the same OCI-only shape that started this investigation. `--raw` said otherwise immediately. Before trusting a tool's negative result, ask what type it's reading the answer into.
- **Two independent confirmations beat one plausible mechanism.** `skopeo`'s silent omission and `docker inspect`'s hard type error are two different tools hitting the same gap two different ways — that's what makes "readers don't carry an OCI-reserved field through" a finding, rather than a guess dressed up as one.
- **A shared symptom across two containers is evidence of a shared cause — it's not evidence of *which* cause.** Both `tor` and `sslh` losing visibility the same way pointed at something systemic; it took checking both the bytes and the reading tool to find out which side of that pipeline the systemic part was actually on.
- **A root cause confirmed doesn't mean the blast radius is what you assumed.** The honest next question after "found it" is "so what actually breaks," not "how do I fix it everywhere."
- **Some findings are correctly closed as "won't fix."** Not every root cause has a repair; some are the visible edge of a default that isn't worth fighting to reverse.
